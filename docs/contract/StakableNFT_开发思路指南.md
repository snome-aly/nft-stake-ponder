# StakableNFT 合约开发思路指南

> 本指南不提供代码实现，而是引导你思考合约设计的各个方面

---

## 🎯 合约目标

**核心定位**：一个可以被质押的 NFT，具有稀有度属性，不同稀有度影响质押奖励

**使用场景**：
- 用户铸造 NFT
- NFT 具有稀有度（Common / Rare / Epic / Legendary）
- 质押池合约需要读取 NFT 的稀有度来计算奖励
- NFT 可以被转移、授权给质押池

---

## 🤔 第一步：核心问题思考

在开始写代码前，先回答这些问题：

### 问题组 1：基础功能
1. **这个 NFT 需要符合哪个标准？**
   - 提示：ERC721 是基础，但有哪些扩展可以用？
   - 思考：为什么需要 Enumerable？为什么需要 URIStorage？

2. **谁可以铸造 NFT？**
   - 提示：是任何人都可以铸造，还是只有特定角色？
   - 思考：如何实现访问控制？
   - 答案：支持任何人铸造，铸造时收取一定费用到合约owner，每人最多铸造20个  总量上限100个
   
3. **铸造时需要传入什么参数？**
   - 思考：接收者地址、稀有度、元数据... 还有别的吗？
   - 答案：铸造数量(numberOfTokens)

### 问题组 2：稀有度设计
4. **如何在链上存储稀有度？**
   - 提示：Solidity 有哪些数据类型适合存储固定的几个选项？
   - 思考：为什么不用 string 存储 "Common"、"Rare"？

5. **每个稀有度对应的奖励倍率是多少？**
   - 提示：Common 1x, Rare 1.5x, Epic 2x, Legendary 3x
   - 思考：如何在合约中表示小数？（Solidity 没有浮点数）

6. **稀有度倍率应该存储在哪里？**
   - 提示：每个 NFT 都存一份？还是全局存一份映射？
   - 思考：哪种方式更省 Gas？

### 问题组 3：数据结构
7. **需要存储哪些 NFT 属性？**
   - 稀有度
   - 铸造时间
   - 还需要其他的吗？

8. **如何组织这些数据？**
   - 提示：Solidity 有 struct（结构体）
   - 思考：用什么作为 key 来映射到这些数据？

### 问题组 4：对外接口
9. **质押池合约需要调用哪些函数？**
   - 查询稀有度？
   - 查询奖励倍率？
   - 转移 NFT？

10. **前端需要调用哪些函数？**
    - 铸造 NFT
    - 查询用户持有的 NFT 列表
    - 查询某个 NFT 的详细信息
    - 查询总供应量

## 🔧 第三步：技术难点思考

### 难点 1：多重继承冲突

**问题**：当你继承多个合约时（如 ERC721 + ERC721Enumerable + ERC721URIStorage），会遇到函数冲突

**需要重写的函数**：
- `_update`
- `_increaseBalance`
- `tokenURI`
- `supportsInterface`

**思考任务**：
1. 为什么这些函数需要重写？
2. 重写时应该调用哪个父合约的函数？
3. 调用顺序重要吗？

**学习建议**：
- 查阅 OpenZeppelin 文档关于"多重继承"的部分
- 理解 C3 线性化算法
- 了解 `override(A, B)` 的语法

---

### 难点 2：稀有度倍率的数值表示

**问题**：Solidity 没有浮点数，如何表示 1.5x？

**几种方案**：
1. 用整数表示百分比（100 = 1.0x, 150 = 1.5x）
2. 用 basis points（10000 = 100%）
3. 直接用整数倍率，调用时需要除以基数

**思考任务**：
- 哪种方案最适合你的需求？
- 如果用方案 1，质押池计算时如何使用这个倍率？
- 精度够用吗？

---

### 难点 3：tokenURI 的元数据管理

**问题**：NFT 的元数据（图片、属性等）存储在哪里？

**几种方案**：
1. 铸造时传入完整的 IPFS CID
2. 合约内部拼接 baseURI + tokenId
3. 动态生成（根据稀有度返回不同 URI）

**思考任务**：
- 对于本项目，哪种方案更简单？
- 如果用方案 1，谁负责上传到 IPFS？
- 元数据 JSON 的格式是什么？

## 🎯 开发建议

### 增量开发策略
1. **先写最简版本**
   - 只实现铸造功能
   - 不考虑稀有度，所有 NFT 都一样
   - 确保能编译和部署

2. **逐步添加功能**
   - 添加稀有度枚举
   - 添加倍率查询
   - 添加元数据存储

3. **不断测试**
   - 每添加一个功能，立即写测试
   - 确保旧功能没有被破坏

### 遇到问题时
1. **编译错误**
   - 仔细阅读错误信息
   - 检查语法（分号、括号）
   - 检查 import 路径
   - 检查函数重写是否正确

2. **逻辑错误**
   - 使用 `console.log` 调试（需要 import hardhat/console.sol）
   - 写更详细的测试
   - 逐行检查状态变化

3. **不理解某个概念**
   - 查阅 OpenZeppelin 文档
   - 查阅 Solidity 官方文档
   - 查看其他项目的实现（如 CryptoKitties）

## 💡 关键思考题

在完成合约后，思考这些问题：

### 设计决策
1. 为什么选择这种数据结构？有没有更好的方案？
2. 如果未来要支持 100 种稀有度，你的设计需要改动多少？
3. 倍率配置是写死在合约里，还是应该支持动态修改？

### Gas 优化
4. 哪些操作最消耗 Gas？
5. 如果要节省 Gas，可以优化哪里？
6. 存储 vs 计算，哪个成本更高？

### 安全性
7. 有没有可能出现整数溢出？（Solidity 0.8+ 自动检查）
8. 有没有重入攻击风险？（这个合约有外部调用吗？）
9. 管理员权限是否过大？

### 可扩展性
10. 如果要添加"升级稀有度"功能，需要修改什么？
11. 如果要支持"NFT 合成"，需要添加什么？
